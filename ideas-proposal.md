# Property-Driven RL for Software Bug Fixing

## Core Idea

Learn to fix bugs through RL guided by formal properties.
SWE-bench Verified as the final benchmark.
Instead of learning "end-to-end issue -> solution", decompose the problem: generate target properties from the issue description, then use RL to search for code modifications that satisfy these properties.
The overall focus here is on generating meaningful, verifiable properties from natural language issues.

## Architecture

Three components:

### 1. Property Generator: Prover Model

- Input: issue description + current code + failing tests
- Output: target properties (pre/postconditions, type constraints)
- Training: supervised learning on SWE-bench PRs
- Baseline: fine-tune [NL2Contract](https://arxiv.org/pdf/2510.12702) model to predict property changes from issues

### 2. Code Modifier (RL Agent)

- State: current code, property satisfaction status, test results
- Action: fine-grained code edits (insert/del/replace lines)
- Policy: learns modifications which move toward satisfying given properties
- Training: online RL with property-based rewards

### 3. Property Verifier (Environment)

- Converts properties to executable checks
- Runs:
  - type checker (mypy)
  - property-based tests using the Python Hypothesis library which are generated by the Prover Model
  - unit tests (from SWE-bench itself)
- Returns: reward signal based on property satisfaction for the Code Modifier (RL Agent)

## Training Setup

### Property Generator

- Dataset: SWE-bench PRs with before/after code
- Apply NL2Contract to both buggy and fixed code to extract properties, then learn the mapping from issue -> property delta
- Task: learn to predict properties that distinguish buggy from fixed code
- Validation: check if generated properties hold in fixed code and are violated in buggy code

### Online RL

- Environment: SWE-bench issues
- agent proposes code modifications iteratively
- verifier checks properties after each modification
- reward: weighted combination of property satisfaction, test pass rate, and regression penalties
- terminate: when all properties satisfied or max. iterations reached

## Goal

Structured feedback: instead of sparse binary reward, agent gets continuous feedback from property satisfaction.

- partial progress is rewarded -> incremental learning
- agent knows which changes help -> better credit assignment
- explicit properties explain what's wrong -> interpretability

## Future

- Static Verification:
  The current approach uses property-based testing (Hypothesis) for pragmatic verification. To get sound verification, we could integrate a static verifier like [Viper](https://www.pm.inf.ethz.ch/research/viper.html)/[Nagini for Python](https://github.com/marcoeilers/nagini).
- Higher-Level Formal Reasoning:
  Properties could also be formalized in theorem provers like Isabelle/HOL or Lean 4 to reason about meta-properties (e.g., "these properties guarantee correctness", "property A implies property B").
  This would enable formal verification of the property decomposition strategy, not just individual code implementations.
